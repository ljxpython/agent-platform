# Platform Web Vue 控制面页面开发标准

这份文档把 `apps/platform-web-vue` 的正式页面范式固定下来。后续页面继续开发，不再回到“先把页面拼出来再说”的散装模式。

---

## 1. 页面结构

每个正式页面默认拆成 4 层：

1. 路由入口
2. 页面壳层
3. service 调用层
4. 复用组件层

固定要求：

- 路由只做参数承接和鉴权进入
- 页面只负责编排展示与交互
- 请求统一从 `src/services/*` 发起
- 复用组件不直接依赖具体业务接口

---

## 2. UI 标准

- 页面必须有稳定标题区
- 列表页优先使用统一筛选区、统计卡、表格区三段结构
- 风险动作必须有确认弹窗
- loading / empty / error 不能省
- 大屏下内容区必须撑满，不允许出现残缺留白
- 深色模式和浅色模式都要保持首屏一致性

---

## 3. 交互标准

- 不自动做让用户迷路的跳转
- 长列表默认保留筛选和分页状态
- 后台轮询或流式更新不能强行抢滚动焦点
- Drawer / Modal 打开后要保证小分辨率可见
- 需要用户确认的动作，不允许只靠回车误触发

---

## 4. Service 标准

- 页面不直接拼接 URL
- 模块请求统一收敛到对应 service
- service 负责接口契约归一化
- 错误解析统一走共享 HTTP 错误处理
- 不允许把 legacy fallback 写回正式 service

---

## 5. State 标准

- 全局登录态放 store
- 当前项目上下文放 store
- 页面筛选、弹窗、临时表单状态优先放页面内部
- 不把一次性弹窗状态塞进全局 store
- 不维护双 token、双项目上下文、双控制面状态

---

## 6. 审计标准

前端虽然不直接写审计，但正式页面必须遵守审计触发点：

- 创建
- 更新
- 删除
- 启停
- 批量操作
- 配置变更

页面开发时至少要能回答：

- 这个动作命中哪个后端接口
- 后端会产出哪个 audit action
- 成功和失败后用户能在哪里看到结果

---

## 7. 新页面最低交付物

- 页面源码
- 对应 service
- 类型定义
- 最小 loading / empty / error
- 手动验收步骤
- 必要时补路由和导航入口

---

## 8. 明确禁止

- 页面直接发裸请求
- 页面偷偷拼 legacy 地址
- 一个页面同时维护两套控制面数据源
- 把调试性质组件直接塞进正式导航
- 为了赶进度省掉空态、错态、确认弹窗
